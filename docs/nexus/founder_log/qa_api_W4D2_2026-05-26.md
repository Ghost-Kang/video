## 3. WebSocket lifecycle

需先发 `{type:"auth",user_id:X}`(否则 close 4001 "未认证"),auth 后自动下发 `session_list`。

| Msg type | Result |
|---|---|
| connect (no auth) | close `4001 未认证` —— 安全行为 PASS |
| `auth` | 收 `session_list` PASS |
| `list_sessions` (无 thread_id) | **NO_RESP**(被 line 506-508 的 `if not thread_id: continue` 静默吞掉) **P2** |
| `list_sessions` (带 thread_id) | 收 `session_list` PASS |
| `get_session_state` | 收 `session_state{messages,canvas}` PASS,自动 ensure_session_exists |
| `rename_session` (已删) | **NO_RESP**,连接存活 → 回归 **PASS** |
| `delete_session` | 无 ACK 但实际生效(list 从 1 → 0) **WARN(P2)** 前端没法确认 |
| 未知 type `this_does_not_exist` | 静默 drop,连接存活 PASS |
| 发 `"{not json"` malformed | **连接立即 close 1011 internal error** **FAIL (P1)** —— `json.loads(raw)` 在 server.py:483 无 try/except |

## 7. 整体健壮性结论

**当前不建议直接承载 10 人并发内测,需先修 #1 / #2 / #3 三个 P1。**

主要风险:
- **P1-1 (WS 崩连)**:任一前端打错一个字符就断连,Phase 1 用户体验直接破裂,且无重连自愈逻辑可见。
- **P1-2 / P1-3 (HTTP 500)**:`event_name` 笔误或前端 JSON 序列化 bug → 5xx 直接污染监控告警,真正的故障会被淹没。
- **P1-4 (零单测覆盖 WS handler)**:`rename_session` 这次删除虽侥幸通过,但下次 server.py 改动无安全网。

修复成本估算: 3 个 P1 都在 server.py 同一个 `try/except` 范围内,**预计 2-3 小时**全部解决。P2 可在 Phase 1 期间滚动修。

修完 P1 后,Phase 1 内测 ≤10 人并发,在 fixture upstream 下系统**可承载**(因为 226 pytest 全过,核心 service 层稳定,问题集中在 server.py 边界层错误处理)。
