## 0. 一页纸 TL;DR（给 founder）

- **当前能验证什么 vs 不能**：线上对用户开放的只有「**分析单环**」（粘 URL→看懂为什么火）。改写被 `REWRITE_ENABLED=false`（`frontend/src/App.tsx:157`）+ 后端 `CASCADE_REWRITE_UPSTREAM` 默认 fixture 暂挂；生成 leg 与分析**解耦**、只能前端手动 `execute_node` 触发、且**零成本护栏**；发布包 `PublishPackCard` **不被任何页面渲染**、是死代码。
- **所以访谈分两类证据**：①**已上线环节**（H1 从热点开始 / H2 为什么火有用）→ 现场让用户真用，采**行为 + 主观**双证据；②**未上线环节**（H3 改写够用 / H5 生成够发布）→ 无法现场跑真产品，采**需求强度 + 原型反应 + 现工作流替代成本**的**前瞻证据**，明确标注「这是意向不是行为」。
- **样本对齐 Gate**：≥10 试用 / ≥5 完成首条（首条 = 跑完一次真 URL 分析并能复述价值）/ ≥3 一周内回访（PHASED_PLAN §4.4）。
- **数据有现成落点**：events 表已内建 `interview_logged`（字段 `value_statement_match / would_pay_39 / notes_url / niche`）和 `consent_accepted` 事件（`backend/src/agent/cascade/events.py:40-41`）——访谈结论直接 `emit` 进同一张 events 表，与 P2-10 埋点漏斗对齐，不另起 CSV。

## 1. 目标与假设映射（每问必须挂一个 Gate 指标或 H 假设）

PHASED_PLAN §7 的 H1-H8 是这轮访谈的脊柱。本轨聚焦 founder 点名的 **H1 / H2 / H3 / H5** + 端到端**需求强度**与**卡点**，顺带采 H4/H6/H7 的弱信号。

| 假设 | 一句话 | 现产品能否现场验证 | 本访谈采的证据类型 | 映射 Gate 指标 |
|---|---|---|---|---|
| **H1** | 用户真的从热点开始（而非自带选题） | ✅ 能（分析入口已上线） | 行为：现场看 ta 上一条内容怎么来的 + 现场粘一条爆款 | 热点→创作转化≥15% |
| **H2** | 浅分析「为什么火」对改写真有用 | ⚠️ 半能（分析能看，改写不能跑） | 主观：看完爆点卡后能否说出「我会改哪一句/哪一镜」 | ≥2 人一句话说出价值 |
| **H3** | 改写产物质量足以让用户继续而非弃稿 | ❌ 不能（改写暂挂） | 前瞻：现工作流里 ta 现在怎么「借鉴」爆款 + 对 fixture/llm 样例的弃稿点 | （Phase 2 改写采用率前置） |
| **H5** | 生成质量足以支撑实际发布 | ❌ 不能（生成解耦+无护栏+不在旅程） | 前瞻：愿不愿意发「AI 生成的镜头图」+ 当前出图/拍摄的真实成本 | 单条<¥15 / 发布率 |
| H4（弱） | 锚点跨 run 复用可感知 | ❌ 不能 | 前瞻：ta 有没有「固定人设/固定场景」复用习惯 | 锚点复用≥2 人 |
| H6（弱） | ¥39/月 模型成立 | ❌ 不能 | 主观：`would_pay_39` 直接问 | （Phase 2 付费） |
| H7（弱） | 失败可恢复时不流失 | ✅ 能（FailureBanner 已在） | 行为：故意喂一条坏 URL/超长视频，看 ta 下一步 | 失败有下一步 100% |

**北极星问题（每场必答，写进 `value_statement_match`）**：
> 访谈结束时，用户能否**用自己的话**说出「它帮我把别人的爆款变成我的内容」这一类价值？（Jobs 视角，Gate 第 4 项；记 `match=true/false/partial`）

## 2. 招募

### 2.1 来源与鉴权
- **从现有邀请码 cohort 招**（两层鉴权：COHORT 层走 `X-Invite-Code`，ADMIN 层走 `X-Admin-Token`，见 memory `http_auth_model`）。每位受访者给一个 cohort 邀请码，**一人一码**便于 events 表按 user_id 归因（注意技术债：`cost_guard` cap key 用客户端传入 user_id，同 cohort 共享码者可绕——访谈期人少可接受，但归因时知道这点）。
- **不公网招募**：避免无身份流量污染 events 漏斗与成本归因。

### 2.2 画像（真实中文短视频/抖音创作者，去 niche 化）
- **必须**：过去 30 天发过 ≥1 条短视频（抖音为主，小红书可），自己做选题或脚本（哪怕很糙）。
- **配额（验证需求强度多样性，非严格分层）**：
  - 「**临摹型**」3-4 人：明确会去抄/借鉴爆款（直接打 H1/H3）。
  - 「**自带选题型**」3-4 人：基本不看爆款、自己定选题（H1 的**反证组**，防确认偏误）。
  - 「**卡在改写型**」2-3 人：想借鉴但不知怎么改成自己的（H3 最相关）。
- **排除**：MCN 批量代运营、纯搬运号、从不自己产出脚本者（不是目标用户，会污染信号）。

### 2.3 样本量与节奏（对齐 Phase 1 Gate）
| Gate 指标 | 招募目标 | 本轨怎么收 |
|---|---|---|
| ≥10 真实试用 | 招 **12** 人（留 2 缓冲） | 每人 1 场 45-60 min 访谈 + 现场真用分析 |
| ≥5 完成首条 | 现场陪跑确保 ≥6 人跑通一次真 URL 分析 | 「首条」= 跑完分析 + 能复述价值（改写未上线，**不能**要求跑完整闭环；这是 Gate 口径偏差，须向 founder 标注） |
| ≥3 一周内回访 | 访谈结束约「一周后回看/再试一条」 | 第 7±2 天发回访（见 §4.4），记 `interview_logged` 第二次 |

> **Gate 口径警示（回灌给 founder 决策）**：PHASED_PLAN §4.4 的「完成第一条内容」原义是跑完**完整闭环**（含改写/草稿/发布包）。当前改写暂挂，本轨只能验证「完成一次分析」。访谈无法替代「完成首条内容」的 Gate，只能为**改写解封后**的完整 Gate 准备基线。这是未决项 §7.3「定位是否把完整闭环作主线」的直接输入。

## 3. 访谈脚本（45-60 min，半结构化 + 情境法）

> **主持铁律**：少说多看（master-apprentice 模型）。开放式优先；**绝不**问引导性的「你会不会用 AI 改写」这种 yes/no，而问「你上次借鉴爆款是怎么操作的，做给我看」。情境法（看真实素材）的信号 > 任何意向陈述。

### 阶段 0 — 暖场 + 知情同意（5 min）
- 自我介绍 + 录屏/录音征得同意（记 `consent_accepted` 事件，字段 `version/accepted_at/documents`，对齐 `events.py:41`）。
- 「今天主要是看你平时怎么做内容，没有标准答案，越真实越好。」

### 阶段 1 — 现工作流（现在怎么做内容）（10 min，主攻 H1）
开放式 + 情境法：
1. 「**把你上一条内容打开给我看**」——看完整路径：选题从哪来？脚本怎么写的？拍/剪用什么？发之前改了几版？（情境法核心，别让 ta 凭记忆讲）
2. 「这一条的**选题是怎么定的**？是你自己想的，还是看到了什么？」（H1 直接证据：自带选题 vs 从热点）
3. 「从有想法到发出去，**哪一步最痛/最花时间**？」（卡点采集，开放式，不预设答案）
4. 「你平时**会不会看别人的爆款**？看完一般会做什么？」（H1 + 引出 H3 现状）

> 采集落点：H1 行为（选题来源占比）、**现工作流卡点清单**（逐字记，§5.2）。

### 阶段 2 — 看到爆款后第一步（10 min，主攻 H1→H2 衔接 + 现场行为）
1. 「假设你刷到一条很火的同领域视频，**很想做一条类似的，你第一步会干嘛？**」（开放式，先听 ta 的天然动作，不提我们的产品）
2. 现场给 ta（或让 ta 自己挑）**一条真实爆款 URL**，「**现在用这个工具粘进去试试**」——现场跑分析（这是 ≥10 试用 / 转化率的行为证据）。
3. 观察：ta 粘的是分享文案还是链接？（入口短链 302 跟随技术债，retro §5——若 ta 用手机 App 链接当场失败，记下来）。

### 阶段 3 — 「为什么火」有没有用（10 min，主攻 H2）
分析结果回来后（当前结果页形态：`AnalysisStatStrip` → `ViralAnalysisCard` 三级层级[钩子/痛点需求/情绪触发/目标人群大字] → 逐幕 `SceneAnalysisCard` + `ScriptDrawer` 抽屉——**不是**抓/留/带三幕，别向用户暗示三幕）：
1. 「**你看到的这些，哪一块对你最有用？哪一块你直接跳过？**」（维度价值排序；低质量分析会出现「未识别XXX」中文兜底占位 `_VIRAL_FALLBACKS`——记 ta 看到占位的反应）。
2. 「看完之后，**如果让你做一条**，你会**改哪一句台词 / 哪一个镜头**？说给我听。」（H2 关键：分析是否真能转成 ta 的改写动作，**这是改写需求的天然引出**）
3. 「『原视频脚本』抽屉你会点开看吗？看分镜还是逐字稿？」（验证 ScriptDrawer 价值）

> 采集落点：H2 主观价值 + **维度有用性排序**（哪些维度可视觉降权/隐藏，回灌 UX）。

### 阶段 4 — 改写到什么程度愿意用（10 min，主攻 H3，前瞻证据）
> 改写未上线，**不能跑真产品**。用「现工作流替代成本 + 样例反应」采前瞻证据，明确标注意向。
1. 「你现在想把别人的爆款变成自己的，**实际是怎么改的**？（开新文档？口述？）这一步花你多久？」（H3 现工作流替代成本——**这是改写需求强度的硬证据**）
2. 给 ta 看**两段对照样例**：A=fixture 模板套娃版（visual_palette 拼接，当前线上版本）、B=doubao llm 代笔版（来自改写质量 agent 的真 URL 样例）。「**这两个版本，哪个你会直接拿去改？哪个你会丢掉？为什么？**」（**弃稿点采集**，§5.3 核心）
3. 「一个改写稿要**满足什么条件**，你才会接着往下做（而不是关掉自己重写）？」（**「足够好的改写」的用户语言定义**——逐字记，回灌质量 agent）
4. 「如果改写稿**第一句不抓人 / 像机器写的 / 有广告法违禁词**，你会怎样？」（标定 confidence 阈值与 scrub 严格度的用户容忍线）

> 采集落点：H3 需求强度（替代成本）+ **弃稿点** + **「足够好」定义**（§5.3，回灌改写质量 agent）。

### 阶段 5 — 愿不愿意用 AI 生成的镜头图发布（8 min，主攻 H5，前瞻证据）
1. 「你现在一条视频的**画面/镜头是怎么来的**？自己拍？找素材？花多少钱多少时间？」（H5 替代成本基线）
2. 给 ta 看 1-2 张 image-grounded 草稿图样例。「**这种 AI 生成的镜头图，你会直接发出去吗？**还是只当参考/分镜？」（H5 关键：发布门槛 vs 参考门槛是两回事）
3. 「什么情况下你**绝对不会**发 AI 图？」（H5 反向边界 + 合规顾虑——注意 IMAGE_GEN_PROVIDER 默认 google 跨境，合规需 PM 确认，但**别向用户解释技术细节**，只采 ta 的顾虑）

> 采集落点：H5「发布门槛」vs「参考门槛」+ 出图替代成本（喂 H6 成本对比）。

### 阶段 6 — 端到端 + 收尾（7 min）
1. 「如果有个工具能**从粘一条爆款链接，一路到帮你出一个能直接发的版本（脚本+镜头图+标题标签）**，这对你**值多少**？现在用什么替代？」（端到端需求强度 + H6）
2. 直接问：「**¥39/月你会付吗？为什么？**」（记 `would_pay_39`）
3. 北极星（§1）：「用你自己的话，这个工具**到底帮你做了什么**？」（记 `value_statement_match`）
4. 「最后——**你最希望它先帮你解决哪一个**：选题、改写、出图、还是发布？」（端到端各 leg 需求强度排序）

### 阶段 7 — 失败恢复旁路（机会触发，主攻 H7）
若现场分析**真的失败/超时**（坏 URL / 超长视频，doubao_direct 单发 120s+S5 重试最坏接近 360s；或 passthrough 模式时长护栏 no-op），不要救场——观察：
- ta 看到 FailureBanner（CardStack.tsx）后**知不知道下一步**？（H7 + Gate「失败有下一步 100%」）
- 时长护栏拒绝文案（「这条 X 秒太长，建议先剪到 3 分钟以内」）ta 能否看懂照做？

## 4. 可量化产出（每场访谈强制收口的结构化数据）

### 4.1 每场必填字段（直接 `emit('interview_logged', ...)` 进 events 表）
对齐 `backend/src/agent/cascade/events.py:40` 的 `INTERVIEW_LOGGED` 必填字段：
- `value_statement_match`: `true` / `partial` / `false`（北极星，Gate 第 4 项）
- `would_pay_39`: `true` / `false` / `unsure`（H6）
- `notes_url`: 本场逐字笔记/录屏链接
- `niche`: 受访者自报领域（去 niche 化后为自由文本，非旧三 niche 枚举）

### 4.2 每场额外采集（写进 notes_url 指向的笔记，结构化）
| 字段 | 取值 | 映射 |
|---|---|---|
| `selection_source` | hot_trend / self_originated / mixed | H1（热点 vs 自带选题占比） |
| `analysis_ran` | true/false（现场是否跑通真 URL） | ≥10 试用 / 转化率 |
| `analysis_useful_dims` | 有用维度排序 + 跳过维度 | H2 + UX 维度降权 |
| `would_change_after_analysis` | 看完能否说出具体改哪句/哪镜 | H2（分析→改写动作转化） |
| `current_rewrite_cost` | 现工作流改写一条耗时（分钟） | H3 替代成本 |
| `discard_points` | 弃稿点（见 §5.3，逐条） | H3 + 回灌质量 agent |
| `good_enough_def` | 「足够好的改写」用户原话 | **回灌质量 agent**（§5.3） |
| `would_publish_ai_image` | publish / reference_only / never | H5 |
| `ai_image_blocker` | 绝不发 AI 图的条件 | H5 边界 |
| `first_leg_to_solve` | 选题/改写/出图/发布 | 端到端需求排序 |
| `failure_next_step_known` | true/false（若触发失败） | H7 / Gate 失败 100% |

### 4.3 跨场聚合产出（≥10 场后出，回灌建设轨）
1. **Gate 收口表**：≥10 试用 / ≥5「完成首条（=跑通分析）」/ ≥3 回访 / ≥2 value_statement_match=true 的真实计数（带 Gate 口径偏差注脚，见 §2.3）。
2. **H1-H3/H5 证据强弱判定**：每假设标 strong / weak / refuted + 支撑引语。
3. **现工作流卡点排名**（§5.2）：按提及频次 + 痛感强度排序，喂 Phase 2 优先级（端到端哪条 leg 先做）。
4. **「足够好的改写」用户语言词表**（§5.3）：回灌改写质量 agent 作为验收标准（不是工程标准，是用户语言）。
5. **弃稿点清单**：fixture vs llm 对照样例的弃稿原因频次（直接定改写解封的质量底线）。

### 4.4 回访（第 7±2 天，喂「一周内回访 ≥3」）
- 异步消息 + 可选 15 min：「这周有没有再想起/再用？做了第二条吗？为什么没用？」
- 记第二次 `interview_logged`（`value_statement_match` 是否变化 = 价值是否留存）。

## 5. 三个回灌建设轨的专门采集

### 5.1 给 PM / 阶段门：Gate 口径偏差
当前无法验证「完成首条**内容**」（改写/生成/发布未对用户开放），只能验证「完成一次**分析**」。访谈结论必须显式写明这道偏差，作为 §7.3「是否把完整闭环作主线」的 founder 决策输入；并对照 retro §6「改写解封 = 第一张多米诺」。

### 5.2 给产品优先级：现工作流卡点清单（逐字 + 频次）
每场阶段 1 问 3「哪一步最痛」逐字记，跨场聚合排名。预期候选（不预设，让用户说）：选题不确定 / 不会拆解爆款 / 改写成自己的太费劲 / 出图素材贵 / 发布标题标签难写。排名决定**端到端四条 leg 谁先做**——直接喂 Phase 2 P2-1（生成）vs 改写解封的优先级。

### 5.3 给改写质量 agent：「足够好的改写」用户语言定义 + 弃稿点
> 这是本轨**对改写解封最关键的产出**。改写质量 agent 当前的硬约束很严（shots<3 或 script_markdown 不在 80-600 字或任何 scrub 触发 → confidence≤0.4），需要用户语言标定阈值是否合理。
- **采「足够好」原话**（阶段 4 问 3），不替用户总结。聚类成可验收的检查项，例如「第一句必须像我自己会说的话」「镜头不能和原片一模一样」「不能有审计员口吻」。
- **采弃稿点**（阶段 4 问 2，fixture vs llm 对照）：哪些缺陷让用户**立刻关掉**（机器味/套娃感/违禁词/与源片雷同/太长）。
- **回灌格式**：交给改写质量 agent 时标注「这是用户语言验收标准，不是工程指标；用于校准 confidence 阈值与 scrub 严格度；改写解封需配合铁律①bump revision，且 rewrite 缓存当前无 revision 守卫（切 llm 后 24h 内会返回旧 fixture 结果，验收时务必清缓存或换 user_id）」。

## 6. 并行节奏（与建设轨并行的运行办法）

> founder 明确「并行的……从跟内测用户交谈」。本轨与建设轨**并行不阻塞**：建设轨继续做改写解封/生成/埋点，本轨同时跑访谈采证据，两轨在 events 表 + 本文档聚合区交汇。

| 环节 | 谁 | 怎么做 | 回流到哪 |
|---|---|---|---|
| 约人 | 本轨主持（founder 或指定）| 从 cohort 邀请码池发码 + 约时间 | 招募名单（私有，含 user_id↔邀请码映射） |

| 结构化收口 | 主持当场填 §4.1/4.2 | 当场 `emit interview_logged` + `consent_accepted` | **events 表**（与 P2-10 同一张，不另起） |

**节奏建议**：第 1 周跑 4-5 场（含 1 场反证组「自带选题型」防确认偏误），出首版 H1/H2 判定；第 2 周补满 ≥10 场 + 启动回访。**不等满 10 场才回流**——每跑 3 场更新一次聚合区，让建设轨实时拿到信号。

## 7. 主持纪律与偏误防范

1. **行为 > 意向**：情境法（看真实素材、现场真用）的信号权重远高于「你会不会用」的口头意向。所有前瞻问题（H3/H5）在聚合时**标注 `intent_only`**，不与行为证据等权。
2. **反证组必招**：3-4 名「自带选题型」专门证伪 H1，避免只招临摹型导致 H1 假性成立。
3. **不卖产品**：访谈不演示愿景、不解释技术（doubao/google/合规/clip best-effort 一律不提）。只采用户怎么想、怎么做。
4. **别暗示三幕**：当前结果页是 toprador 维度 + 三级层级 + 脚本抽屉，**不是**抓/留/带三幕（retro §2.2 反转②）。介绍分析结果时用中性语言。
5. **失败不救场**：现场出错时观察用户下一步（H7），不要立刻帮 ta 修——这是 Gate「失败有下一步 100%」的唯一真实采集机会。
6. **样例要诚实**：阶段 4 给的 fixture vs llm 对照、阶段 5 的草稿图，必须是**真 URL 跑出来的**真实样例（找改写质量 agent / 生成轨要），不能用挑过的最佳样例制造假信号。
